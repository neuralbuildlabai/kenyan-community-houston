# Production Bootstrap — First Super Admin

This is the only supported, repo-tracked path to make
`admin@kenyancommunityhouston.org` the first `super_admin` on a
brand-new production Supabase project. It is intentionally split
into two stages so neither stage requires the repo to hold a
password or service-role key.

> **Stop.** Do not run any of this against staging or against an
> existing production project that already has a super_admin.
> The bootstrap script below is one-shot.

---

## Stage 1 — Create the auth user (Supabase Dashboard)

This stage is manual on purpose. The repo never sees the
temporary password.

1. Open the Supabase Dashboard for the **production** project.
2. Auth → Users → **Invite User** (or **Add user → Create new
   user**).
3. Enter:
   - Email: `admin@kenyancommunityhouston.org`
   - Mark "Email confirmed" so the user can sign in immediately.
   - Set a strong temporary password (≥ 16 chars, generated by
     a password manager). Copy it to 1Password under
     "KIGH production / first super admin (one-time)".
4. Save.
5. Confirm the new row exists in `auth.users` with
   `email_confirmed_at IS NOT NULL`.

Do **not** edit `profiles` from the dashboard. The next stage
does that with the correct invariants and writes the
`admin_user_profiles` row that the password-rotation gate uses
on first sign-in.

---

## Stage 2 — Promote the profile to super_admin

This stage is automated by
`scripts/admin/bootstrap-super-admin.mjs`. The script:

- refuses to run unless `KIGH_ENV=production`;
- refuses to run unless `KIGH_PRODUCTION_CONFIRM=YES_I_AM_BOOTSTRAPPING_PRODUCTION`;
- refuses to run if `SUPABASE_URL` matches an entry in
  `KIGH_STAGING_SUPABASE_URLS` (defence-in-depth: avoids
  bootstrapping a super_admin into the staging project by
  mistake);
- refuses to overwrite an existing super_admin in the same
  project unless `KIGH_BOOTSTRAP_OVERWRITE=YES_OVERWRITE_EXISTING`
  is set explicitly;
- writes:
  1. `profiles.role = 'super_admin'` for that auth user
     (creates the row if missing);
  2. `admin_user_profiles.must_change_password = true` with
     `temporary_password_set_at = now()`, so the first sign-in
     forces a rotation.

Run it like this (export each variable in the shell — never put
secrets in command history if you can avoid it):

```bash
export KIGH_ENV=production
export KIGH_PRODUCTION_CONFIRM=YES_I_AM_BOOTSTRAPPING_PRODUCTION
export SUPABASE_URL="https://<production-ref>.supabase.co"
export SUPABASE_SERVICE_ROLE_KEY="<paste-once-from-1password>"
# Optional belt-and-braces:
export KIGH_STAGING_SUPABASE_URLS="https://<staging-ref>.supabase.co"
export KIGH_PRODUCTION_PROJECT_REF="<production-ref>"

node scripts/admin/bootstrap-super-admin.mjs admin@kenyancommunityhouston.org
```

Expected output ends with:
```
✅ admin@kenyancommunityhouston.org is now super_admin in production.
   Sign in once at /admin/login. The change-password gate will
   require a new password before the dashboard loads.
```

---

## Stage 3 — Verify

1. In a private window, navigate to the production URL → `/admin/login`.
2. Sign in with the temporary password from Stage 1.
3. The app must redirect to `/admin/change-password`. Set a new,
   strong password.
4. After rotation, the app must redirect to `/admin/dashboard`
   and the sidebar must show `System Health` (because
   `super_admin` is in `SYSTEM_HEALTH_ADMIN_ROLES`).
5. From `/admin/users`, confirm:
   - the role picker contains `super_admin`, `platform_admin`,
     and the operational roles;
   - the page lists the freshly-bootstrapped account with role
     `super_admin`.
6. Delete or rotate the Stage 1 temporary password from
   1Password once Stage 3 completes successfully.

## SQL fallback (only if Stage 2 fails)

If for some reason the script cannot run (e.g. the host doesn't
have Node available) and the operator must do this in Supabase
SQL editor, the equivalent statements are:

```sql
-- Replace <UID> with the auth.users.id of admin@kenyancommunityhouston.org
update public.profiles
   set role = 'super_admin',
       email = 'admin@kenyancommunityhouston.org',
       updated_at = now()
 where id = '<UID>';

-- If profiles.id row doesn't exist yet:
insert into public.profiles (id, email, role)
select id, email, 'super_admin'
  from auth.users
 where email = 'admin@kenyancommunityhouston.org'
on conflict (id) do nothing;

-- Force a password rotation on first sign-in
insert into public.admin_user_profiles (
  user_id, must_change_password, temporary_password_set_at,
  password_changed_at
)
select id, true, now(), null
  from auth.users
 where email = 'admin@kenyancommunityhouston.org'
on conflict (user_id) do update
   set must_change_password = excluded.must_change_password,
       temporary_password_set_at = excluded.temporary_password_set_at,
       password_changed_at = null,
       updated_at = now();
```

Run that block once. Confirm:

```sql
select p.email, p.role, aup.must_change_password
  from public.profiles p
  left join public.admin_user_profiles aup on aup.user_id = p.id
 where p.email = 'admin@kenyancommunityhouston.org';
```

should return one row with `role = 'super_admin'` and
`must_change_password = true`.

---

## Things this path explicitly does NOT do

- Does not create the auth user. That stays in the Supabase
  Dashboard so the password is never typed into the repo.
- Does not seed any other admin accounts. Use the
  `create-admin-user` Edge Function from the Admin Users page
  after `super_admin` is signed in.
- Does not run UAT helpers
  (`scripts/admin/create-uat-users.mjs` refuses to target
  production by guard).
- Does not write to staging.
